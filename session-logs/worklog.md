License Apache-2.0, very active (5,396 commits, multiple per day), Kotlin 98%.
- Each extension = one Kotlin class extending `AnimeHttpSource` (or `AnimeSourceFactory`/a theme base) implementing ~6 abstract methods (`popularAnimeRequest/Parse`, `searchAnimeRequest/Parse`, `animeDetailsParse`, `episodeListParse`, `videoListParse`/`getVideoList`). HTTP via OkHttp (provided by host app), HTML via Jsoup, JSON via kotlinx.serialization. Configurable mirrors/servers via `ConfigurableAnimeSource` + `SharedPreferences`.
- The video URL is almost never in the catalog HTML; it's hidden inside an embed/iframe "video server" page and extracted by one of ~10 patterns documented with real code snippets: direct JSON field, inline `<script>` string concat (StreamTape), MD5+token+random-suffix (Dood), AES-CBC encrypted AJAX with key-in-CSS-classes (GogoStream), Dean-Edwards packed JS + regex (MP4Upload, Kwik HLS), packed JS + custom base-N cipher + POST form + 302 redirect (Kwik MP4), Synchrony-deobfuscated main.js + DMCA server rotation (StreamWish), GraphQL + AES-GCM + XOR (AllAnime), and the universal WebView-request-interception fallback (`Regex(".*\\.(mp4|m3u8|mpd)(\\?.*)?$")`).

CI runs `aapt dump badging` + an "inspector" job that DexClassLoads each APK to query the source class, emits `index.min.json` to the `yuzono/anime-repo` repo branch, host app fetches the index, downloads APKs on demand.

All do the same thing: WebView loads URL → JS auto-clicks Turnstile → poll `CookieManager` for `cf_clearance=` → copy cookies to OkHttp jar → retry.
- For a homegrown scraping browser: (1) define a `Site` interface with ~8 methods, (2) steal `lib/universalextractor`'s WebView-request-interception as the universal fallback, (3) steal `lib/playlistutils` for HLS master playlist parsing, (4) steal the Cloudflare WebView-cookie-poll pattern, (5) plan for per-host extractors with hardcoded keys/secrets, (6) prefer theme inheritance for sites sharing a CMS, (7) the standalone-APK + DexClassLoader packaging is overkill — a flat folder of site modules loaded at startup is simpler.

2. **The "universal WebView interceptor" (task 2-b's `lib/universalextractor` regex `.*\.(mp4|m3u8|mpd)(\?.*)?$`) is necessary but NOT sufficient.** It works as a fallback / last resort for 3 of 5 sites (anikototv, animepahe, miruro) once CF is solved — but mkissa's AES-GCM+XOR source encryption is specifically designed to defeat it, and reanime's auth wall prevents the player from even loading the stream without a session. 3. **AniList GraphQL (https://graphql.anilist.co) is the universal catalog source for 4 of 5 sites** (anikototv uses anilist.co for poster/banner art; mkissa pulls metadata via AllAnime which itself proxies AniList; reanime uses AniList for cover/banner + Jikan for episodes; miruro's secure pipe wraps AniList queries). 4. **The theme-extractor pattern (task 2-b's AnikotoTheme) is the highest-leverage code in the entire ecosystem** — it covers anikototv + aniwave + any future Anikoto-CMS site with one base class + a 1-line subclass. Reverb should port this verbatim as `lib-multisrc/anikototheme` equivalent.
5. **Miruro's `/api/secure/pipe` is the most elegant aggregator pattern observed** — a single base64+gzip+encrypted tunnel that proxies multiple providers server-side, returning direct m3u8 URLs. The reverse-engineered `walterwhite-69/Miruro-API` is a complete reference implementation worth studying.
6. **No site uses DASH (.mpd).** All streams are HLS (.m3u8) or progressive (.mp4). Media3's HlsMediaSource covers everything; no DashMediaSource needed.
7. **5 of 5 sites have NSFW/banner ad inventory** (typical for the genre). Reverb's content-filter / ad-blocker layer must run on every page load (EasyList-style cosmetic filtering + network-level ad-domain blocklist) — this is a baseline UX requirement, not an afterthought.
8. **Auth-gated sites (reanime) need an account-management subsystem** in Reverb that the user didn't explicitly ask for but is implied: storing per-site login cookies, presenting a login WebView when 401 is hit, persisting credentials. Should be added to the Phase 1 roadmap.

**For Phase 0 spike (the ≥7/10-sites gate from PLAN.md):** of these 5 sites, 3 (anikototv, animepahe, miruro) should yield a playable stream via the universal interceptor + CloudflareBypass combo; 2 (mkissa, reanime) will require additional per-site work and may fail the spike. Recommend the Phase 0 spike test list include 2-3 anikototv + 2 aniwave + 1 animepahe + 1 miruro episode URLs as the "high-confidence" subset to clear the 7/10 gate, with mkissa/reanime deferred to Phase 1.

- **The HTML pipeline is the key innovation, not the model size.** Tested against real anikototv.to HTML: 97 KB raw → 2.6 KB compact (~650 tokens) by detecting the most-repeated card pattern and emitting only `<title>` + nav snippet + 3 sample cards. - **Prompt + validation pseudocode + 3-call flow are in §3.3-§3.6 of the report.** Three LLM calls per site (catalog, details, episode), each with up to 3 retries that feed validation failures back to the model. Total per site: ~3-8 sec with Groq remote, ~20-60 sec on-device, ~0 sec manual.
